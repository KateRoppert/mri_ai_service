# Stage 01: Protocol-Specific Series Scoring — Design Spec

**Дата:** 2026-06-26
**Ветка:** `feat/prod-readiness`
**Этап:** 6 (Production Readiness v1), приоритет 1
**Закрывает:** KI-027, KI-001

## Проблема

`SeriesDeduplicator._select_best_series` (`scripts/01_reorganize_folders.py`)
выбирает серию для модальности по единственному критерию — максимум
`slice_count`. В клинических архивах для одной модальности в одной сессии
часто конкурируют несколько серий: 3D-оригинал и его 2D MPR-реформаты,
несколько базовых протоколов (TSE vs TFE), серии с артефактами движения,
и иногда серии другой анатомии (шейный отдел позвоночника), которые
текущий `ModalityDetector` не отличает от мозговых.

Связанная проблема KI-001: FLAIR-серии с реально присутствующим контрастным
маркером логируются с `contrast=True`, что выглядит как баг, хотя
постконтрастный FLAIR — обычный клинический случай.

### Подтверждение на реальных данных

`data/MS_5/P000915`, обе сессии (`2022-01-18`, `2023-03-25`) дают точное
воспроизведение проблемы:

- **T1c:** 6 конкурирующих серий — `*_CE_T1-TSE_(3D_brain)` +
  2 его MPR-реформата (cor/axi, 2.5mm), и `*_CE_T1-TFE_(3D_brain)` +
  2 его MPR-реформата. Два разных базовых протокола контрастного T1.
- **FLAIR:** `*_FLAIR_(3D_brain)` (нужный, 3D) vs два
  `*_MPR_FLAIR_2.5mm_(...)` (производные 2D-реформаты).
- **Анатомия:** `*_T2-TSE_(sag_C-spine)` и `*_CE_T1-TSE_(sag_C-spine)` —
  шейный отдел позвоночника, не мозг. Без анатомического фильтра T1c-серия
  C-spine технически проходит паттерн-матчинг (`ce` + `t1`) и попадает в
  кандидаты на мозговую T1c-серию.

### Референс из истории веток

В `origin/refactoring` (коммит `a58fdea`, нояб. 2025) — наиболее зрелая
версия системы скоринга серий, потомок `75ec5b6` (`origin/decomposition`).
Веса (`tfe`/`tse`/`3d`/`mpr`, штрафы за анатомию `spine`/`cervical`/`pit`)
явно подгонялись под протоколы вида `MPR_2.5mm`, `3D_brain` — то есть под
данные, похожие на `data/MS_5`. Взято как отправная точка для весов в новом
конфиге; будет доуточняться при интеграции на реальном прогоне.

Важное ограничение легаси-версии, не переносим как есть:
- скоринг только текстовый (protocol name + series description), не
  учитывает реальное разрешение/толщину срезов из DICOM — а KI-027 явно
  требует это учитывать;
- штраф за анатомию (`spine: 0.8`) мягкий, а не исключающий — то есть
  spine-серия теоретически может выиграть при отсутствии других кандидатов.
  В этом дизайне анатомия — жёсткий фильтр (см. "Анатомическое исключение"
  ниже), а не штраф.

## Архитектура

```
configs/series_scoring.yaml          ← новый конфиг (веса, штрафы, анатомия)
utils/config_loader.py               ← + load_series_scoring_config()
scripts/01_reorganize_folders.py:
  SeriesInfo                         ← + технические поля (slice_thickness_mm,
                                        ti_ms, has_contrast)
  ModalityDetector.detect_modality   ← возвращает (modality, desc, tech_meta)
    + _is_excluded_anatomy()         ← новый метод, жёсткий фильтр
    + KI-001: фикс лога для FLAIR + contrast=True
  SeriesDeduplicator
    + score_series()                 ← новая функция скоринга
    _select_best_series()            ← переписан на скоринг
```

**Поток данных:** `ModalityDetector.detect_modality()` читает DICOM серии
один раз (как и сейчас) и дополнительно достаёт `SliceThickness` (0018,0050)
и `InversionTime` (0018,0082) — без повторного чтения файла. Анатомическая
проверка выполняется до паттерн-матчинга модальности.

Детекция модальности (`_match_modality`, регэксп `_has_ce_marker` и т.д.)
**не меняется** — она уже корректно решает задачу классификации и тщательно
вычищена (защита от ложного "ce" в "space"/"sequence"). Доработка касается
только: (1) что отбрасывается до классификации (анатомия), и (2) как
выбирается лучшая серия среди уже классифицированных кандидатов одной
модальности.

### Анатомическое исключение — жёсткое, без fallback

Если `protocol_name + series_description` содержит маркер не-мозговой
анатомии (`spine`, `cervical`, `pituitary`, `orbit`, `neck`) —
`detect_modality` возвращает `(None, desc, {})`, как если бы паттерн
вообще не сматчился. Серия не попадает в `series_list` в
`_process_one_patient_core` (существующий код это уже корректно
обрабатывает — изменений там не требуется).

**Без отката на анатомически неверную серию, даже если она единственный
кандидат для модальности.** Обоснование: подстановка спинального снимка
вместо мозгового — это не "более низкое качество, но рабочий вариант", а
структурно неверные данные. Дальше по пайплайну это уйдёт на регистрацию к
мозговому атласу и в сегментационную модель — в лучшем случае регистрация
упадёт, в худшем — модель тихо выдаст мусорную "сегментацию" на снимке
шейного отдела, и это попадёт в клинический отчёт как находка по мозгу.
Правильная реакция на отсутствие нужной модальности — то, что уже умеет
`CompletenessChecker` (помечает сессию неполной), а не молчаливая
подстановка неподходящих данных.

### Скоринг кандидатов одной модальности

Применяется только когда у модальности больше одного кандидата (при одном
кандидате — текущее поведение без изменений, как и сейчас в
`deduplicate_session`).

```
score = Π(text_weights, по совпавшим подстрокам в "protocol|series_desc")
        × failure_penalty (если есть маркер failed/repeat/motion/...)
        × resolution_factor(slice_thickness_mm)
        × ti_bonus (только для t2fl, если InversionTime выше порога)
```

где `resolution_factor`:

```
resolution_factor = clamp(
    reference_slice_thickness_mm / actual_slice_thickness_mm,
    min_factor, max_factor
)
```

— тоньше срез относительно эталона → множитель выше (бонус), толще →
ниже (штраф), но не более чем в `min_factor`/`max_factor` раз, чтобы
один артефактный тег не давал экстремальный перекос итогового счёта.
Если `actual_slice_thickness_mm` не прочитался — `resolution_factor = 1.0`
(нейтрально).

Если ни один сигнал не сработал (пустой текст, теги не прочитались) —
все кандидаты получают `score = 1.0`, и тогда применяется текущий
`max(slice_count)` как последний tie-breaker — safety net, не регрессия
для датасетов без насыщенных описаний серий (напр. UPENN-GBM).

## Конфигурация: `configs/series_scoring.yaml`

```yaml
anatomy_exclude:
  keywords: [spine, cervical, c-spine, pituitary, orbit, neck]

failure_markers:
  keywords: [failed, repeat, redo, retry, motion, artifact, incomplete]
  penalty: 0.5

text_weights:
  t1:   {tfe: 3.0, tse: 2.0, se: 1.0, "3d": 1.5, mpr: 0.1, clear: 1.2, brain: 1.1}
  t1c:  {tfe: 3.0, tse: 2.0, se: 1.0, "3d": 1.5, mpr: 0.1, brain: 1.1,
         gad: 1.3, "+c": 1.3, contrast: 1.2}
  t2:   {tse: 2.0, "3d": 1.3, mpr: 0.1, sense: 1.3, brain: 1.1}
  t2fl: {"3d": 2.0, mpr: 0.1, sense: 1.3, long: 1.2, brain: 1.1}

resolution_scoring:
  reference_slice_thickness_mm: 1.0
  min_factor: 0.5
  max_factor: 2.0

flair_ti_bonus:
  threshold_ms: 1500
  bonus: 1.2
```

Веса взяты из `a58fdea` как отправная точка, будут донастраиваться при
интеграции на реальном прогоне `data/MS_5`.

`load_series_scoring_config()` в `utils/config_loader.py` следует тому же
паттерну, что и существующий `load_lesion_type_config()`: читает YAML,
бросает `ConfigValidationError` при отсутствии/повреждении файла — без
тихих fallback-ов на уровне конфига. Загружается один раз в точке
конструирования `ModalityDetector`/`SeriesDeduplicator` (в `main()` /
`run_sequential()` / `process_single_patient()` — там же, где сегодня
конструируются эти классы), а не на каждую серию.

### Решение про lesion-type-специфичность

Пункт роадмапа "для МС — предпочтение 3D FLAIR над 2D" **не требует**
отдельного override по `lesion_type`: универсальные веса `3d: 2.0` /
`mpr: 0.1` в `text_weights.t2fl` уже разрешают этот случай для любого
типа поражения (подтверждено на `data/MS_5`, где конфликт 3D vs MPR FLAIR
воспроизводится буквально).

## KI-001: фикс лога

В `detect_modality`, при `modality == 't2fl' and has_contrast == True`,
заменить текущую тревожную формулировку лога на информационную (контраст
после FLAIR — обычный клинический случай, не используется против
классификации FLAIR). Сам контрастный маркер в тексте уже даёт обычный
бонус в `text_weights.t1c` при скоринге T1c-кандидатов — отдельная
"confidence-система" не нужна.

## Обработка ошибок

- Отсутствует/повреждён `series_scoring.yaml` → `ConfigValidationError`,
  пайплайн падает с понятной ошибкой (как и сегодня при отсутствии
  `lesion_types.yaml`) — без тихого отката на старое поведение.
- Отсутствует отдельный DICOM-тег (`SliceThickness`/`InversionTime`) →
  нейтральный множитель `1.0` для этого сигнала — не ошибка, тег может
  просто не быть записан сканером.
- Анатомическое исключение оставляет модальность без кандидатов → штатно
  обрабатывается существующим `CompletenessChecker` (сессия помечается
  неполной).

## Совместимость

`copy_series()` / `_extract_and_save_metadata()` (анонимизация,
извлечение метаданных) запускаются после дедупликации, только для уже
выбранной серии — этот дизайн меняет лишь то, какая серия выбирается,
сам механизм анонимизации/метаданных не трогается.

- "Confidence-тиры" для источника контрастного маркера (DICOM-тег vs
