# Project Roadmap

Крупноблочный план развития сервиса. Каждый этап — отдельная ветка
и отдельный чат с ассистентом для сохранения контекста.

Текущее состояние: завершены Этапы 1-5 и 5.6 (MS Clinical Report v2),
смержены в main. Этап 5.5 (skull stripping research) в работе отдельной
веткой. Следующий шаг — Этап 6 (Production Readiness).

## ✅ Этапы 1–3: MAS-инфраструктура (завершено)

**Ветка:** `feat/mas-refactor`
**Период:** май 2026
**Статус:** ✅ Готово к merge в main

**Этап 1 — `ServiceBase` для gbm-seg.** Унификация контракта между
сервисами. Извлечение общей инфраструктуры (HTTP, queue, GPU pool)
в `services/common/service_base.py`. gbm-seg переписан как наследник
`ServiceBase`.

**Этап 2 — Путь-ориентированный контракт `/predict`.** Замена legacy
multipart upload на JSON-payload с путями файлов через shared volume.
Подпапки `{lesion_type}/` для multi-model coexistence. Удаление legacy
endpoints (−503 строки).

**Этап 3 — ms-seg сервис.** Второй inference-сервис (МС-сегментация
через nnUNet v2 + CATMIL fork). Реестр сервисов (`configs/services.yaml`).
Параметризация `lesion_type` через всю цепочку: frontend → backend →
orchestrator → pipeline stages. End-to-end verified на реальном
MS-клиническом кейсе.

## ✅ Этап 5: MS Adaptation

**Ветка:** `feat/lesion-type-aware-pipeline`
**Зависит от:** Этапа 4

Сейчас MS-pipeline работает "по краям" — мы прогнали кейс end-to-end,
но не валидировали все этапы детально и не оптимизировали под не-глио
семантику. Этот этап делает MS-pipeline first-class citizen.

**Содержание:**

- **Stage 04 (QA):** lesion-type-aware required modalities
  (см. KI-004)
- **Stage 05 (preprocessing):** skip T1c registration для МС
  (экономит ~25% времени, см. KI-005)
- **Клинический отчёт:** адаптировать метрики под МС (количество
  очагов, локализация, longitudinal analysis между сессиями).
  Требует обсуждения с клиницистами какие метрики критичны.
- **configs/lesion_types.yaml:** config-driven описание модальностей
  и параметров для каждого типа поражения (см. KI-016)
- **Frontend:** корректное отображение МС везде (визуализация маски,
  фильтры, выпадающие списки)

**Цель:** MS равноправен с глио во всех точках pipeline и UI.

## □ Этап 5.5: Skull Stripping Research

**Ветка:** `feat/skull-stripping-research`
**Зависит от:** Этапа 5
**Тип:** Research / optimization
**Статус:** ⏳ В работе (design spec готов), не блокирует Этап 6

Сравнительное исследование инструментов skull stripping в рамках пайплайна.
Текущий инструмент (FSL BET) подобран эмпирически; для гетерогенных данных
(глио + МС, разные центры, разные протоколы) он может не быть оптимальным.

**Содержание:**

- **Отбор инструментов:** кандидаты — FSL BET (текущий), HD-BET, SynthStrip
  (FreeSurfer), ANTs BrainExtraction, optiBET. Критерии отбора: открытый
  доступ, поддержка CPU/GPU, совместимость с пайплайном.
- **Методология:**
  - Прогон каждого инструмента на датасетах глио (UPENN-GBM) и МС (SibBMS)
  - Подбор параметров для каждого инструмента
  - Замер: скорость, потребление RAM/GPU, воспроизводимость
  - Качество: визуальная оценка + метрики (DSC если есть ground truth)
- **Вывод:** выбор оптимального и универсального skull stripper для
  гетерогенных данных; рекомендации по параметрам
- **Интеграция:** замена или параметризация Stage 05 под выбранный инструмент

**Цель:** обоснованный выбор skull stripper, улучшение quality Stage 05
на разнородных клинических данных.

## ✅ Этап 5.6: MS Clinical Report v2 (Вариант C)

**Ветка:** `feat/ms-clinical-metrics`
**Период:** июнь 2026
**Зависит от:** Этапа 5
**Статус:** ✅ Завершено, смержено в main (`c9279c9`)

Второй заход на МС-отчёт с полноценными клиническими метриками.
Вариант A (реализован в Этапе 5) дал базовый отчёт — нагрузка, объёмы,
категории. Вариант C добавляет то, что реально меняет ведение пациента.

**Реализовано:**

- **McDonald-критерии локализации** (Plan 2a/2b): классификация очагов на
  перивентрикулярные / юкстакортикальные / инфратенториальные / спинной
  мозг. `MSZoneAnalyzer` в Stage 08 + API (`/api/mcdonald-reports/{run_id}`)
  + UI-секция в клиническом отчёте + Kappa round-trip.
- **Детекция новых/растущих/разрешившихся очагов** между сессиями
  (Plan 3): `backend/lesion_diff.py` (overlap-based matching с допуском
  `dilation_voxels`) + API (`/api/longitudinal/{patient_id}/diff`) +
  колонка в лонгитюдной панели.
- Попутный фикс: GBM-формы `volume_report` больше не утекают в
  Kappa-карточки MS-сессий.

**Закрыло:** KI-019 (клинический отчёт для МС).

**Сознательно не входило в объём** (зафиксировано в design spec как
deferred, не баги): spinal cord зона (нет spine-регистрации в пайплайне);
гадолиний-накапливающие очаги (ждёт решения по T1c-протоколу для МС);
growth-threshold — временный дефолт, ждёт подтверждения клиницистов.
Известный остаточный issue — недетерминизм счёта очагов из-за самой
модели сегментации (KI-037), смягчён настройкой `dilation_voxels`, но
не устранён на уровне модели.

**Цель:** МС-отчёт, полностью соответствующий клиническому workflow
(мониторинг прогрессии, оценка ответа на терапию). Достигнуто.

**Зависит от:** Этапов 4, 5, 5.6 (5.5 желателен, но не блокирует)

**Содержание (по приоритету):**

   работой через confidence-скоринг в `ModalityDetector`.
2. **Kappa dedup (KI-035):** content-aware update при повторном прогоне
   пациента — обновлять entity вместо skip, если входные данные полнее.
3. Покрытие тестами (unit + integration для критических путей)
4. Документация: README, Architecture overview, API docs, Deployment guide
5. CI/CD pipeline (GitHub Actions): автотест + автодеплой
6. Логирование и мониторинг (Prometheus + Grafana, structured logs)
7. Безопасность: CORS, rate limiting, auth review
8. Backup strategy для БД, runtime configs, mapping files
9. Performance benchmarking baseline
10. Тестовый прогон на больших датасетах (50+ пациентов)
11. UX-фиксы из KI: KI-026, KI-036, KI-040, KI-041

**Цель:** v1.0 release. Сервис готов к развёртыванию в боевой среде
для двух типов поражений (глио + МС).

## □ Этап 7: MAS Implementation (research)

**Ветка:** `feat/mas-coordinator`
**Зависит от:** v1.0 в продакшене
**Тип:** Research-level задача

Это **переход от MAS-ready инфраструктуры к настоящему MAS**.

Инфраструктура, заложенная в Этапах 1-3, поддерживает многосервисную
архитектуру с диспетчеризацией по lesion_type. Теперь добавляем
реальную агентскую логику.

**Содержание:**

- **Coordinator-агент:** LLM-based principal, который анализирует кейс
  и решает какие модели использовать. Использует Kappa, manifest'ы
  сервисов, описание кейса.
- **Quality-агент:** оценивает confidence результата сегментации.
- **Resource-агент:** управляет GPU pool, очередью задач.
- **Реализация cascade/fallback/ensemble режимов** (твоя идея):
  - Cascade: primary → confidence низкая → fallback model
  - Fallback: primary → ошибка → автопереключение
  - Ensemble: N моделей параллельно → консенсус
- **Contract Net Protocol** для агент-сервис коммуникации.
- **Multi-trainer dispatch в ms-seg:** использование всех 11 trainer'ов
  через `options.trainer`.
- **BDI-агенты** (опционально, если research-направление это требует).
- **Benchmark для MAS-метрик:** межагентная коммуникация, latency,
  consensus time.

**Связь с PhD-research:**
Этот этап — основа двух research-направлений:
1. **Функциональные диагностические агенты** (cascade/ensemble)
2. **BDI/Contract Net для ресурсов**

**Цель:** v2.0 с настоящей MAS-архитектурой.

## Принципы работы

**Одна ветка = один чат.** В начале каждого нового этапа создаём
новый чат с ассистентом и передаём контекст (стартовое сообщение
готовится заранее).

**KNOWN_ISSUES.md** — живой документ. Новые баги, найденные в любой
ветке, фиксируются туда. После закрытия — помечаются как закрытые
или удаляются.

**SPEC.md** — отражает текущую архитектуру (не roadmap). Обновляется
по мере завершения этапов.

**Атомарные коммиты с conventional commits.** Каждый коммит — одна
завершённая мысль, проверяется py_compile + end-to-end test перед
push.
